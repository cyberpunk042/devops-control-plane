# Chat Services

> Git-backed messaging embedded in the DevOps control plane.
> Threads live on the ledger branch. Run-attached messages live in git notes.
> All storage is local-first, sync is explicit push/pull.

### Storage Backends

| Backend | What | Where |
|---------|------|-------|
| **Git notes** | Run-attached messages | `refs/notes/chat` on the run's annotated tag |
| **Ledger branch** | Thread messages | `.ledger/chat/threads/<id>/messages.jsonl` |
| **Ledger branch** | Thread metadata | `.ledger/chat/threads/<id>/thread.json` |

### Message Routing

```
send_message(text, run_id=..., thread_id=...)
    ‚îÇ
    ‚îú‚îÄ‚îÄ run_id only      ‚Üí git notes on tag
    ‚îú‚îÄ‚îÄ thread_id only   ‚Üí thread messages.jsonl
    ‚îú‚îÄ‚îÄ both             ‚Üí git notes + thread messages.jsonl
    ‚îî‚îÄ‚îÄ neither          ‚Üí "general" thread (auto-created)
```

### `chat_crypto.py` ‚Äî Message Encryption (140 lines)

AES-256-GCM encryption using the same primitives as `vault.py`.

| Function | Signature | Purpose |
|----------|-----------|---------|
| `encrypt_text` | `(plaintext: str, project_root: Path) ‚Üí str` | Encrypt ‚Üí `ENC:v1:<salt>:<iv>:<ct>` |
| `decrypt_text` | `(enc_text: str, project_root: Path) ‚Üí str` | Decrypt `ENC:v1:...` ‚Üí plaintext |
| `is_encrypted` | `(text: str) ‚Üí bool` | Check `ENC:v1:` prefix |

#### Crypto Parameters

| Parameter | Value | Purpose |
|-----------|-------|---------|
| Algorithm | AES-256-GCM | Authenticated encryption |
| KDF | PBKDF2-SHA256 | Key derivation from passphrase |
| Iterations | 100,000 | Brute-force resistance |
| Salt | 16 bytes (random) | Unique per encryption |
| IV (nonce) | 12 bytes (random) | Unique per encryption |
| Key | 32 bytes | Derived from passphrase + salt |

#### Encrypted Text Format

```
ENC:v1:<base64-salt>:<base64-iv>:<base64-ciphertext+tag>
```

- **Salt** is random per encryption, preventing rainbow table attacks.
- **IV** is random per encryption, ensuring ciphertext is unique even for identical plaintext.
- **Tag** is appended to ciphertext by GCM mode for authentication.

#### Key Source

Reads `CONTENT_VAULT_ENC_KEY` from the project's `.env` file.
Reuses the same key as the content vault file encryption.
Raises `ValueError` if the key is not set.

#### Internal Functions

| Function | Purpose |
|----------|---------|
| `_get_enc_key(project_root)` | Read passphrase from `.env` |
| `_derive_key(passphrase, salt)` | PBKDF2-SHA256 key derivation |

### `chat_ops.py` ‚Äî Core Operations (732 lines)

The main CRUD + sync module for the chat system.

#### Public API ‚Äî Message Operations

##### `send_message`

```python
send_message(
    project_root: Path,
    text: str,
    *,
    user: str = "",
    thread_id: str | None = None,
    run_id: str | None = None,
    trace_id: str | None = None,
    publish: bool = False,
    encrypt: bool = False,
    source: str = "manual",
) -> ChatMessage
```

Sends a message with automatic routing based on provided IDs.

**Routing logic:**
1. If `run_id` set ‚Üí write JSONL to git notes on the run's annotated tag
2. If `thread_id` set ‚Üí append to thread's `messages.jsonl`
3. If both set ‚Üí write to both locations
4. If neither set ‚Üí auto-create/use the `"general"` thread

**Side effects:**
- Ensures ledger worktree exists
- Resolves user from git config if blank
- Parses @-references from original text (before encryption)
- Publishes `chat:message` event to event bus

##### `list_messages`

```python
list_messages(
    project_root: Path,
    *,
    run_id: str | None = None,
    thread_id: str | None = None,
    n: int = 50,
) -> list[ChatMessage]
```

Returns messages newest-first. If both `run_id` and `thread_id` are
provided, merges and deduplicates by message ID. Auto-decrypts if
`CONTENT_VAULT_ENC_KEY` is available; otherwise replaces encrypted
text with `"[encrypted]"`.

##### `delete_message`

```python
delete_message(
    project_root: Path,
    *,
    thread_id: str,
    message_id: str,
) -> bool
```

Rewrites the thread's `messages.jsonl` without the target message.
Returns `True` if found and removed. Publishes `chat:message_deleted`.

##### `delete_thread`

```python
delete_thread(
    project_root: Path,
    *,
    thread_id: str,
) -> bool
```

Removes the entire thread directory from the ledger worktree.
Raises `ValueError` if attempting to delete the `"general"` thread.
Triggers a background push after deletion.
Publishes `chat:thread_deleted`.

##### `update_message_flags`

```python
update_message_flags(
    project_root: Path,
    *,
    thread_id: str,
    message_id: str,
    publish: bool | None = None,
    encrypt: bool | None = None,
) -> ChatMessage | None
```

Toggle publish and/or encrypt flags. When toggling encrypt:
- `encrypt=True` ‚Üí encrypts the text field (if not already encrypted)
- `encrypt=False` ‚Üí decrypts the text field (if currently encrypted)

Returns the updated `ChatMessage` or `None` if not found.
Publishes `chat:message_updated`.

#### Public API ‚Äî Thread Operations

##### `create_thread`

```python
create_thread(
    project_root: Path,
    title: str,
    *,
    user: str = "",
    anchor_run: str | None = None,
    tags: list[str] | None = None,
) -> Thread
```

Creates a new thread. Writes `thread.json` to the ledger branch and
commits. The `thread_id` is auto-generated.

##### `list_threads`

```python
list_threads(project_root: Path) -> list[Thread]
```

Lists all threads from `.ledger/chat/threads/*/thread.json`.
Sorted newest-first by `created_at`.

#### Public API ‚Äî Sync

##### `push_chat`

```python
push_chat(project_root: Path) -> bool
```

Pushes two things:
1. `refs/notes/chat` (run-attached messages) ‚Äî direct `git push`
2. Ledger branch (thread files) ‚Äî delegates to `ledger_ops.push_ledger()`

Returns `True` if both succeed. Treats "everything up-to-date" as success.

##### `pull_chat`

```python
pull_chat(project_root: Path) -> bool
```

Fetches two things:
1. `refs/notes/chat` ‚Äî `git fetch origin refs/notes/chat:refs/notes/chat`
2. Ledger branch ‚Äî delegates to `ledger_ops.pull_ledger()`

Returns `True` if both succeed. Treats "remote ref not found" as success
(notes may not exist on remote yet).

### `refs_parse.py` ‚Äî Reference Parsing (109 lines)

Zero-I/O string extraction layer. Used by `chat_ops.py` (to extract refs
from sent messages) and by `refs_resolve.py`/`refs_autocomplete.py`.

| Export | Type | Purpose |
|--------|------|---------|
| `parse_refs(text)` | `‚Üí list[str]` | Extract `@type:id` references from text |
| `parse_ref_parts(ref)` | `‚Üí tuple[str, str] | None` | Split `@run:run_123` ‚Üí `("run", "run_123")` |
| `_REF_PATTERN` | `re.Pattern` | Compiled regex for matching |
| `_VALID_TYPES` | `frozenset[str]` | 12 valid reference types |
| `_relative_time(iso_str)` | `‚Üí str` | Time-ago formatter |

#### Reference Pattern

```
@<type>:<id>
```

- **Type** must be one of 12 valid types (see below)
- **ID** is `[A-Za-z0-9_\-/.]+` (alphanumeric + underscore + dash + dot + slash)

#### Valid Reference Types (12)

| Type | Example ID | Entity |
|------|-----------|--------|
| `run` | `run_20260217T120000Z_detect_a1b2` | DevOps run |
| `thread` | `thread_20260217T120000Z_c3d4` | Chat thread |
| `trace` | `trace_20260217T120000Z_e5f6` | Trace recording |
| `user` | `JohnDoe` | Git author |
| `commit` | `a1b2c3d` | Git commit (short hash) |
| `branch` | `feature/audit-fix` | Git branch |
| `audit` | `scan_20260217T120000Z_docker` | Saved audit snapshot |
| `code` | `src/core/services/chat/chat_ops.py` | Code file path |
| `doc` | `content/docs/ARCHITECTURE.md` | Document in content vault |
| `media` | `content/images/logo.png` | Media in content vault |
| `release` | `content-vault/ADAPTERS.md` | Release asset (tag/asset) |
| `file` | `content/scripts/setup.sh` | Any file in content vault |

#### `_relative_time` Helper

Converts ISO timestamps to human-readable relative strings:

| Input | Output |
|-------|--------|
| 30 seconds ago | `"just now"` |
| 5 minutes ago | `"5m ago"` |
| 3 hours ago | `"3h ago"` |
| 1 day ago | `"1 day ago"` |
| 14 days ago | `"14 days ago"` |

### `refs_resolve.py` ‚Äî Entity Resolution (335 lines)

Given a parsed reference, looks up the actual entity and returns metadata.

#### Public API

```python
resolve_ref(ref: str, project_root: Path) -> dict | None
```

Returns a dict with at minimum `{"type": ..., "id": ..., "exists": True/False}`.
Returns `None` for invalid references.

#### Resolution Details by Type

| Type | Resolver | Data Source | Return Fields |
|------|----------|-------------|---------------|
| `run` | `_resolve_run` | `run_tracker.get_run_local()` | run_type, subtype, summary, status, started_at, ended_at, duration_ms, code_ref |
| `thread` | `_resolve_thread` | `chat_ops.list_threads()` | title, created_at, created_by |
| `trace` | `_resolve_trace` | `trace_recorder.get_trace()` | name, classification, started_at, auto_summary |
| `commit` | `_resolve_commit` | `git log -1 --format=...` | hash, short_hash, message, author, date |
| `branch` | `_resolve_branch` | `git rev-parse --verify refs/heads/<name>` | sha |
| `audit` | `_resolve_audit` | 3-tier: ledger ‚Üí pending ‚Üí ndjson | source, operation_type, status, timestamp |
| `release` | `_resolve_release` | `release_sync.list_release_assets()` | tag, asset_name, size, download_url, local_path |
| `code` | `_resolve_code` | `Path.is_file()` + stat | size_bytes, path |
| `user` | *(inline)* | No lookup needed | name |

#### Audit Resolution ‚Äî 3-Tier Fallback

```
1. Saved scan snapshots on ledger branch (.ledger/audits/)
   ‚îî‚îÄ‚îÄ source: "ledger", includes full snapshot data
2. Pending (unsaved) audit snapshots (in-memory staging)
   ‚îî‚îÄ‚îÄ source: "pending", includes snapshot data
3. Execution log (.state/audit.ndjson)
   ‚îî‚îÄ‚îÄ source: "ndjson", includes operation_type, status
```

Each tier is tried in order. First match wins.

#### Release Resolution ‚Äî Two Formats

| ref_id Format | Behavior |
|--------------|----------|
| `tag_name` (no slash) | Returns release-level info (tag, asset_count) |
| `tag_name/asset_name` | Returns specific asset info (size, URL, local_path) |

For specific assets, also scans for local copies via `.release.json`
sidecar files in the project tree.

#### Utility

| Function | Purpose |
|----------|---------|
| `_format_size(size_bytes)` | `‚Üí "1.2 MB"` human-readable file size |

## Design Decisions

### Why Git Notes for Run Messages?

Run messages are attached to the run's annotated tag object. Git notes
are the natural mechanism for adding metadata to existing git objects
without modifying history. This means:
- Messages travel with the tag (push/fetch)
- No branch mutation needed for run-only messages
- Messages survive rebases (tags are stable)

### Why Ledger Branch for Threads?

Threads are conversation containers that need directory structure
(thread.json + messages.jsonl). The ledger branch provides:
- File-based storage with git versioning
- Worktree isolation (no main-branch pollution)
- Standard push/pull sync

### Why Two Storage Backends?

Run messages and thread messages serve different purposes:
- **Run messages** are ephemeral commentary on specific operations ‚Äî they
  belong with the run's tag
- **Thread messages** are persistent conversations ‚Äî they belong in the
  ledger's file tree

The `send_message` routing logic handles both transparently.

### Why The `"general"` Thread?

Messages sent without a `run_id` or `thread_id` default to the `"general"`
thread. This provides a catch-all for unanchored messages while keeping the
routing logic simple.

### Why Split `chat_refs.py`?

The original file was 1,281 lines covering three distinct responsibilities
(parsing, resolution, autocomplete). The split gives:
- `refs_parse.py` (109 lines) ‚Äî pure string operations, zero I/O
- `refs_resolve.py` (335 lines) ‚Äî entity lookup with lazy service imports
- `refs_autocomplete.py` (885 lines) ‚Äî rich suggestion generation

The backward-compat shim (`chat_refs.py`, 41 lines) means zero consumer
changes were required ‚Äî all 37 test imports, 2 internal imports, and 4
route imports continue to work unchanged.

### Why Not Split `chat_ops.py`?

At 732 lines, `chat_ops.py` is a single coherent module covering the full
message/thread lifecycle. Splitting it would create artificial boundaries
between tightly coupled operations (e.g., `send_message` depends on
`_ensure_general_thread`, `_append_thread_message`, `_tag_object_sha`,
and `_parse_notes_messages`). The file is under the 1,000-line threshold.

### Why Same Key as Content Vault?

Chat encryption reuses `CONTENT_VAULT_ENC_KEY` because:
- Users already have this key configured for file encryption
- One passphrase to manage, not two
- Chat messages often reference vault content, so same trust boundary
